# Docs-Only Architecture Elements

## Purpose

This document captures architectural and pedagogical elements that are already present in MilEd.One documentation and should be integrated into the current architecture picture.

This is a docs-only synthesis.

It does not rely on git history, commit narratives, or implementation reality.

## Main Conclusion

The current architecture work is missing several important layers that already exist in the documentation corpus.

The most important missing additions are:

1. phase-based governance
2. runtime enforcement architecture
3. full binding-principles implementation logic
4. classroom integration continuum
5. pedagogical DNA taxonomy
6. effort regulation model
7. continuity and opening rituals
8. population routing and differentiated pedagogy

These are not optional enrichments.

They are part of the theoretical architecture of the system.

## Canonical Elements To Incorporate

## 1. Phase-Based Runtime Governance

### What it is

A six-phase enforcement model in which pedagogical rules shift by context of learning, not only by bot identity.

### Why it matters

It adds a missing dimension to the architecture:

- diagnostic
- development
- reflection
- design
- teaching development
- analytics

This means the same engine can behave differently because the pedagogical phase changes, not because an entirely different product is invoked.

## What This Changes In The Current Picture

The current architecture picture should no longer be described only as a map of spaces.

It should become a layered architecture with at least these levels:

1. Kernel / constitution
2. Runtime governance and enforcement
3. Phase-based pedagogical regime
4. Branches: courses / skills / faculty / admin
5. Differentiation and population routing
6. Course and skill planning surfaces
7. Lesson runtime and classroom orchestration
8. Continuity, evidence, and redesign

## Final Recommendation

The next synthesis document should explicitly absorb these documentation-based elements.

Without them, the architecture remains too interface-oriented and not sufficiently faithful to MilEd's documented theoretical system.
