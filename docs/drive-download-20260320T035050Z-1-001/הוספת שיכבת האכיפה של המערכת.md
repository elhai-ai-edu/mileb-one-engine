# **3️⃣ פירוק ה-Guards לפי העקרונות שלך**

## **🔹 Language Gate**

### **Pre-Guard:**

- זיהוי שפת קלט
- בדיקת מצב LANGUAGE_MODE
- אם בקשה לא תואמת מדיניות → חסימה

### **Post-Guard:**

- זיהוי שפת פלט
- אם הפלט אינו בשפת תוצר מוגדרת → שליחה חוזרת לתיקון

## **🔹 Never-Do Principle**

### **Pre-Guard:**

Regex / Pattern Detection:

- “כתוב לי את העבודה”
- “תפתור במקומי”
- “תן לי תשובה מלאה”

אם מזוהה:
→ לא שולחים למודל
→ מפעילים Refusal Engine

זה חייב לקרות לפני קריאה ל-LLM.

## **🔹 No-Skip (Stage Gate)**

אתה צריך לשמור State Object לכל שיחה:

conversationState = {

   currentStage: "analysis",

   readinessEvidence: false

}

אם user מבקש לעבור שלב:
→ בודקים readinessEvidence
→ אם false → חסימה והחזרה לשלב

זה כבר State Machine אמיתי.

## **🔹 Agency Preservation**

### **Post-Model Guard:**

אם הפלט מכיל:

- תשובה מלאה
- ניסוח מסכם סופי
- פתרון מלא

אתה מזהה זאת (heuristic או rule-based)
ומחליף בגרסה Scaffolded.

## **🔹 Uncertainty Gate**

אם המודל עונה בצורה מוחלטת על שאלה שאינה בתחום KB מוגדר:
→ הפעלת Confidence Check
→ הוספת Qualifier אוטומטי

## **5) מימוש “מינימום שעובד” ל־3 עקרונות מחייבים (לקראת סדנה)**

### **A) Never-Do (Request Pattern Blocker) — Hard Fence**

לפני המודל:
- מזהה ניסיונות “כתוב לי”, “פתור לי”, “תן תשובה מלאה”
- לא קורא למודל בכלל
- מחזיר “סירוב פדגוגי + חלופה”

**דוגמת חוקים (Regex)**

- /כתוב( לי)?/
- /פתור( לי)?/
- /תן( לי)? תשובה( מלאה)?/
- /עשה( זאת)? בשבילי/

(אפשר לדייק אח״כ, אבל זה עובד.)

### **B) Language Gate — Runtime Language Validator**

יש לך בשאלון “עברית בלבד / שפת אם לחשיבה + עברית לתוצר”.

מימוש מינימלי:

- Post-Guard בודק: האם הפלט בעברית כשנדרש “output_hebrew”.
- אם לא: מבקש תיקון מיידי (קריאה נוספת למודל עם “rewrite to Hebrew”), או מחזיר הודעת תיקון.

**בדיקת שפה מינימלית (heuristic)**

- אחוז תווים בטווח עברי \u0590-\u05FF
- אם מתחת לסף (למשל 30%) → לא עברית.

זה לא מושלם, אבל מספיק כ־Gate ראשוני אמיתי.

### **C) No-Skip — Stage Gate Validator (State Machine מינימלי)**

אתה צריך State קטן לכל שיחה:

- currentStage
- readiness (true/false או מדורג)
- evidenceNotes (קצר)

בגלל שאין לך DB חובה בשלב זה, הכי פרקטי לסדנה:

- ה־UI שולח state בכל בקשה
- ה־Runtime מאמת אותו עם חתימה (כדי שלא יזייפו)
- ה־Runtime מחזיר state חתום חדש

**למה חתימה?
**כי אחרת סטודנט יוכל לשלוח {"currentStage":"final","ready":true} ולעקוף את ה־Gate.

פתרון מינימלי:

- stateToken = HMAC(stateJson, SECRET)
- כל בקשה מגיעה עם stateJson + stateToken
- ה־Function מאמת ומחליט אם לאפשר מעבר שלב

זה הופך את “No-Skip” ל־Enforcement אמיתי.

**שאלה I3 – רמת דרישות אקדמיות בקורס
** מה רמת הדרישות האקדמיות שאתה מצפה לה?
 ⬜ בסיסית – בעיקר הבנה ותרגול יסודי
 ⬜ בינונית – דיוק מושגי ומבנה תקין
 ⬜ מתקדמת – טיעון מורכב, שילוב מקורות, עומק
 ⬜ גבוהה במיוחד – סטנדרט מחקרי/דיסציפלינרי גבוה

**שאלה I4 – מעקב ושמירת נתונים
** האם חשוב לשמור תוצרים/היסטוריה לצורך למידה ומעקב?
 ⬜ לא – אין שמירת מידע מעבר לשיחה
 ⬜ כן – שמירת תוצרים בלבד (ללא טקסט שיחה מלא)
 ⬜ כן – שמירת תהליך מלא לצורך מחקר/הערכה (בכפוף להסכמה)

# **2) השכבה הנסתרת: Schema מערכתית מלאה (מה שיופק מאחורי הקלעים)**

המרצה רואה רק את השאלות הפדגוגיות.
 המערכת מפיקה אוטומטית SPI JSON עם שדות כמו:

{

 "botType": "writing",

 "interactionMode": "short|process|hybrid",

 "languagePolicy": "HEB_ONLY|L1_THINK_HEB_OUTPUT|CUSTOM",

 "evaluationPolicy": "none|formative|criteria_only|summative",

 "rubricStatus": "provided|partial|none",

 "academicLevel": "basic|intermediate|advanced|high",

 "_layer": "surface|structured|deep",

 "ceiling": "low|mid|high",

 "dataTracking": "none|artifacts_only|full_process"

}

זו בדיוק המשמעות של “C – שכבה גלויה ושכבה נסתרת”.
