# Architecture Canon And Shared Libraries

## Purpose

This document is a short canon.

Its role is to stop a recurring source of drift in MilEd.One:

- architecture absorbs too much
- shared knowledge remains too vague
- product flows carry knowledge that should live above them

This canon fixes that by declaring two things clearly:

1. what stays in core architecture
2. which shared libraries now deserve first-class status below architecture and above products

## Core Rule

Architecture must contain only what the whole platform must obey.

Shared libraries must contain what many products, branches, and bots must reuse.

Everything else should remain in product protocols, bot packs, or content assets.

## Part I: Core Architecture Canon

The following belong in core architecture.

They are not optional libraries.

They are platform law, shared contracts, or runtime boundaries.

### 1. Constitutional Rules

- One-Engine Principle
- immutable kernel rules
- mandatory system-level constraints on bot behavior

### 2. Runtime Governance

- phase-based governance
- enforcement pipeline
- readiness and escalation rules
- evidence-based confirmation rules

### 3. Shared System Contracts

- branch model and branch relations
- Lesson Space as runtime meeting layer
- continuity object boundary
- evidence object boundary
- course-skill bridge contract
- credential object boundary when formalized

### 4. Ownership And Layer Boundaries

- what belongs to planning
- what belongs to runtime
- what belongs to orchestration
- what belongs to redesign

### 5. Core Identity Contracts

- operation mode roles
- variable-class contract where system-wide
- bot construction constraints that protect one-engine integrity

## What Does Not Belong In Core Architecture

The following are too important to ignore, but not correct as architecture law:

- active-learning pattern catalogs
- skill-diagnosis frameworks as content-rich libraries
- transformation templates
- credential recipes
- bot-specific pedagogical methods
- domain-specific skill progressions

These should not be promoted into the core canon unless they become system-wide contracts.

## Part II: First-Class Shared Libraries

These libraries are now important enough to be named as platform structures.

They are not architecture law, but they are too central to remain buried in raw docs or single bots.

### Library 1: Skill Ontology And Diagnosis Library

### Role

Defines how the system names, groups, diagnoses, and evidences skills.

### Why it is first-class

It underpins:

- Skills Branch
- course-skill bridges
- lesson evidence
- credentials
- dashboards
- skill-oriented bots

### Library 2: Transformation Template Library

### Role

Holds reusable redesign patterns for courses, units, and lessons.

### Why it is first-class

It underpins:

- lecturer redesign products
- course planning
- mini-course generation
- active lesson redesign

### Library 3: Active Learning And Skills-Oriented Design Pattern Library

### Role

Holds the underlying pedagogical patterns beneath transformation workflows.

### Why it is first-class

It prevents every product or bot from hardcoding its own teaching patterns.

### Library 4: Product Protocol Registry

### Role

Holds canonical descriptions of major product flows.

### Why it is first-class

It connects architecture to actual operations.

Without it, reusable knowledge remains disconnected from product behavior.

### Library 5: Credential Logic Library

### Role

Holds reusable models of skill-to-evidence-to-credential progression.

### Why it is first-class

It supports stackable learning outputs without pushing every credential detail into architecture.

## Secondary Shared Libraries

These matter, but should follow after the first wave.

- Population Routing And Adaptation Library
- Evidence And Continuity Object Library
- Bot Archetype And DNA Registry

## Part III: Promotion Rule

No material should move from library level into architecture unless it satisfies at least one of these:

- cross-system governance requirement
- shared runtime-state requirement
- shared interoperability contract
- platform-wide enforcement dependency

This rule prevents architecture inflation.

## Part IV: Immediate Consequence

Stabilizing architecture now does not mean writing more broad system overviews.

It means doing two things in order:

1. preserve the core canon as small and strict
2. extract the highest-value shared libraries from the document corpus

## Final Statement

MilEd.One should now be stabilized through a two-tier model:

- a strict core architecture canon
- a growing but governed layer of shared platform libraries

That is the right balance between rigidity and growth.
